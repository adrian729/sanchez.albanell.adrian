# cv-sync: keep future/final_cv.md a 1:1 mirror of the CV page

`templates/cv.html` is the CV: a Jinja fragment included by `templates/index.html` (the web SPA, CV at `#/cv`) and `templates/cv_pdf.html` (the PDF, `/cv.pdf`, WeasyPrint). `future/final_cv.md` is a plain-markdown mirror of the same information for convenient reading and editing; the app never reads it.

These two files are the only content sources and targets. `dist/` (generated by `scripts/build-static.py`), `future/original_cv.md`, and any PDFs are never read as sources and never edited.

## Direction rule

- **Default, always: page → markdown.** Auto-runs after a page edit, bare `/cv-sync`, and any ambiguous request ("sync the CV") all go this way.
- **markdown → page only when the user's current request explicitly says so** ("I updated the markdown, update the page", `/cv-sync to-page`). A to-page request from an earlier turn does not carry forward to a later sync. If the phrasing leaves any doubt about direction ("update the CV page"), ask before touching the page.

## Modes

| Invocation | Behavior |
|---|---|
| `/cv-sync` (no args), or after a page edit | Sync page → markdown |
| `/cv-sync check`, or "are these in sync?" | Report per-section information differences. Edit nothing. |
| `/cv-sync to-page`, or an explicit current request | Sync markdown → page |

"Information differences" means wording, dates, numbers, URLs, presence and order of entries. The per-medium formatting and the Contact ordering described below are not differences; do not report or "fix" them.

## Page → markdown

1. Read `templates/cv.html` and `future/final_cv.md`.
2. Guard against clobbering manual work: run `git diff -- future/final_cv.md` and classify the uncommitted changes line by line (a single hunk can mix both kinds). Added lines whose information is already on the page are sync leftovers or a parallel edit of both files; ignore them. Added lines whose information is NOT on the page, and deleted lines (a deliberate removal), are manual authoring: show the user those exact lines and ask whether to discard them (page wins) or apply them to the page first (their answer counts as the explicit to-page request). Never silently overwrite a manual markdown edit without showing its text.
3. Walk the section mapping. For each section, compare the page's text content (tags, icons, and separators stripped) with the markdown.
4. Make the markdown match the page exactly on information: every sentence and bullet verbatim, dates, numbers, URLs (the href, not the visible link text), and order. Order is recursive: section entries, bullets within a work card, and items in every list. One exception: the Contact list keeps the markdown's own grouping (see conventions); match its set of entries and their text, not its order.
5. Keep the markdown's section order (its established linearization of the page's three print pages) and its formatting conventions.
6. Page content with no markdown home: insert it at the position mirroring the linearization the markdown already uses for its page neighbors (e.g. a new page-2 sidebar section lands near Education and Background), as `##` for a new page section or `###` under an existing one, following existing conventions. Note in the summary that you chose the placement.
7. Markdown content absent from the page: delete it and call the deletion out in the summary.
8. If nothing would change, edit nothing and say so.
9. Finish with a per-section summary: "in sync" or one line on what changed.

## Markdown → page (explicit request only)

1. Confirm the direction is explicit (Direction rule), then compare the same way with the markdown as driver.
2. Edit only text content in `templates/cv.html`. Preserve all markup verbatim: classes, Jinja icon macros, `cv-page` article boundaries, ` · ` separators, the download anchor. Map emphasis back: markdown `**…**`/`_…_` become `font-semibold`/`font-medium` spans and `<em>`. The `.cv-back-btn` lives in `templates/index.html`, intentionally outside the fragment; never move it into `cv.html`.
3. New entries or sections: clone the closest existing block (e.g. a whole `cv-card`) rather than inventing markup.
4. The three `cv-page` articles are fixed print pages, and WeasyPrint pagination cannot be predicted from text length. Never reflow content across them; after any to-page edit, recommend checking `#/cv` and `/cv.pdf`.
5. If nothing would change, edit nothing and say so.

## Section mapping

| `templates/cv.html` | `future/final_cv.md` |
|---|---|
| Header: `cv-name`, `cv-title`, `cv-profile` | `# <name> — CV` (the H1 embeds the name; update it together with `**Name:**`), `**Title:**`, `**Profile:**` |
| Contact (`cv-contact-list`, page 1 sidebar) | `## Contact` bullets |
| Skills (`cv-skill-row`s, page 1 sidebar) | `## Skills` fenced code block, same row order |
| Spoken Languages (page 2 sidebar) | `### Spoken Languages`, nested under `## Skills` even though it is a sibling section on the page; intentional, do not flag or move it |
| The 5 work `cv-card`s, page order across all three pages | `## Work Experience`, one `###` per card, same order |
| Education (page 2 sidebar) | `## Education` |
| Background (page 2 sidebar) | `## Background` |
| Beyond Work (page 3 main; `cv-role` paragraphs "Music", "Electronics for music") | `## Beyond Work` with `### Music`, `### Electronics for music` |
| Personal Learning (page 3 sidebar) | `## Personal Learning` |

## Formatting conventions (information identical, formatting per medium)

- Tech lines: page separates with ` · `, markdown with `, `.
- Work entry headers: page `cv-role` "Role — Company" plus a `cv-role-location` span; markdown `### Role — Company (Location)`. Copy the location string verbatim (some cards carry ` · hybrid / remote`, some have no work mode); never add or remove one. Meta line: page calendar icon, dates, `·`, `cv-meta-tag`; markdown `**dates** — _tag_`.
- Education and Background do not use the work-entry convention: Education is `### Degree — School, City` then `**dates** — Specialization`; Background maps page paragraphs to markdown bullets. Match the text, not the block type.
- Contact: markdown uses bold labels and markdown links; the page uses icons and anchors. URLs are the information and must match exactly; visible anchor text may differ per medium (the page shows `portfolio`, the markdown shows the identifier). The remote-first line is an unlabeled italic note on both sides (`text-cv-muted italic` ↔ a plain `_…_` bullet). The two files deliberately group contact items differently; keep each file's own order.
- Skills block: the markdown code block pads label columns for alignment; keep the alignment when editing.
- Emphasis: `<em>` ↔ `_…_`; `font-semibold`/`font-medium` spans ↔ `**…**`.

## Never

- Never edit `templates/cv.html` outside an explicit to-page request.
- Never edit `dist/`, `future/original_cv.md`, the `{{ pdf_filename }}` anchor, or `app/__init__.py` (the PDF filename is app config, not CV content, even though it contains the name).
- Never reword, "improve", or summarize content while syncing in either direction. This skill copies information; it does not edit prose.
